Большинство движков баз данных SQL (каждый движок базы данных SQL, кроме SQLite, Насколько нам известно) использует статическое, жесткое печатание. При статическом наборе текста, тип данных значение определяется его контейнером - конкретной колонкой в которые хранится значение.

SQLite использует более общую систему динамического типа. В SQLite - тип данных Значение связано с самой величиной, а не с ее контейнером. Система динамического типа SQLite является обратной совместим с более распространенными системами статического типа других движков баз данных в том смысле, что SQL-указания, работающие на статически набранных базах данных Работайте так же в SQLite. Однако динамическая типизация в SQLite позволяет делать вещи, которые невозможны в традиционном жестком типе Базы данных. [Гибкая типизация - это особенность](https://sqlite.org/flextypegood.html) SQLite, а не ошибка.

Обновление: По состоянию на версию 3.37.0 (2021-11-27), SQLite предоставляет [таблицы STRICT](https://sqlite.org/stricttables.html) которые делают жесткий тип правоприменения для разработчиков, которые предпочитают такие вещи.



Каждое значение, хранящееся в базе данных SQLite (или манипулируется движок базы данных) имеет один из следующих классов хранения:

- **НУЛЛ.** Значение является значением NULL.
    
- **ЧИСХЕР.** Значение представляет собой подписанное целое число, хранящееся в 0, 1, 2, 3, 4, 6 или 8 байт в зависимости от величины значения.
    
- **РЕАЛЬНО.** Значение является значением с плавающей точкой, хранящимся как 8-байтное число плавающей точки IEEE.
    
- **ТЕКСТ.** Значение представляет собой текстовую строку, хранящуюся с использованием кодирование базы данных (UTF-8, UTF-16BE или UTF-16LE).
    
- БЛОБ**.** Значение представляет собой каплю данных, хранящихся точно так же, как Это был вклад.

Класс хранения является более общим, чем тип данных. Класс хранения INTEGER, например, включает в себя 7 различных целочисленных типов данных разной длины. Это имеет значение для диска. Но как только целочисленные значения считываются с диска и попадают в память для обработки, они преобразуются в самый общий тип данных (8-байтовое целое число со знаком). Таким образом, по большей части термин «класс хранения» неотличим от термина «тип данных», и эти два термина можно использовать как взаимозаменяемые. Любой столбец в базе данных SQLite версии 3, за исключением столбца INTEGER PRIMARY KEY, может быть использован для хранения значения любого класса хранения. Все значения в SQL-операторах, будь то литералы, внедренные в текст SQL-операторов, или параметры, привязанные к предварительно скомпилированным SQL-операторам, имеют неявный класс хранения. В описанных ниже обстоятельствах ядро СУБД может преобразовывать значения между числовыми классами хранения (INTEGER и REAL) и TEXT во время выполнения запроса.

Жестко типизированная база данных преобразует строку '123' в целое число 123, а целое число 456 - в строку '456' перед выполнением вставки. Для обеспечения максимальной совместимости между SQLite и другими ядрами СУБД, а также для того, чтобы приведенный выше пример работал на SQLite так же, как и на других ядрах СУБД SQL, SQLite поддерживает концепцию "сходства типов" для столбцов. Сходство типов столбца является рекомендуемым типом для данных, хранящихся в этом столбце. Важная идея здесь заключается в том, что тип является рекомендуемым, а не обязательным. В любом столбце можно хранить данные любого типа. Просто некоторые столбцы, получив выбор, предпочтут использовать один класс хранения, а не другой. Предпочтительный класс хранения для столбца называется его «сходством».
