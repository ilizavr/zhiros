#AftonJIT_allocation
# my plan for making JIT-allocator to the future

  **I've thought allocation strategy to the future**. It works like MCB in MS DOS,but Since AftonJIT working wit' high-level virtual memory it has one significant difference. Here's explanation:

- jit_alloc() function takes one argument "count of JIT-pages",JIT-page - one memory region at AftonJIT with size 1 KB. For example, if jit_alloc takes 3, it means "allocate three JIT-pages"
- but, when the first call goes, firstly it allocates the real virtual memory via mmap with flags PROT_WRITE | PROT_READ  probarly wit' size 4 MB
- every block of JIT-page has metadata struct like MCB in start. These blocks help allocator(and external agent too) grab metrics,see gaps,freeing,etc..
- if the current virtual memory(mmap) not enough for allocating JIT-pages,jit_alloc rollbacks to allocate new mmap region in 2 times bigger than past region.
- But it physical memory is gone,ONLY THEN jit_alloc not  takes arguments and end proggram

Keep in mind,that plan is **prototype** but prototype born robust.

Why i'm use the strategy like MS-DOS? because in 12 years i've written prototype of allocator based on MCB strategy. That prototype not working,but,it build mental map of allocating strategy in my mind


#V8_code_positions
# positions of significant tools in V8 code
- assembler_ia32.cc:659 - pc_ variable
- assembler_ia32.cc:653 - buffer_ variable 
- assembler_ia32.cc:1961 - GrowBuffer function that makes new buffer size
- assembler_ia32.h:724 - EnsureSpace class for ensuring current buffer may to take enough binary code
- globals.h:265 - CodeDesc structure
- relocation - is a term at V8/JIT context that means **fixing of addres-stubs  to real in generated code after get knowlege about real location of machine code**
- synopsys about Assembler class constructor - it takes two arguments:
```cpp
// buffer - pointer used foe writing machine code
// buffer_size - its size
Assembler(void* buffer, int buffer_size);
```

1. If buffer is NULL - Then Assembler uses its own internal buffer, growing it. buffer_size argument determines the initial buffer size. Upon destruction of Assembler object, buffer also deallocating
2. If buffer isn't NULL - Then Assembler uses the provided buffer for code generation. buffer_size also determines initial size. Upon destruction of Assembler that buffer not deallocating - you need to do it manually. If buffer too smal,fatal error occurs.
- How i'm realized,pc and buffer uses for separate purposes. pc uses for putting machine code,buffer uses to move code from pc and execute it from the buffer
- last_pc_ field at Assembler class used for saving the last addres of pc before calling new emitter
