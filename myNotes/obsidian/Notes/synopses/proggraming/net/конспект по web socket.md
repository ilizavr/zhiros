# зачем нужен

представьте - вы тимлид который получил сообщение от начальства о том что проект нужно перевести в режим микросервисов. вы ждете когда ваши разработчики спросят вас о новостях и только тогда отправляете письмо 

начальника, вы так делаете с каждым участником вашей команды. согласитесь - это меддлено и добавляет кучу накладных расходов на ожидание, и вы решаете отправить сообщение всем участникам независимо от того,  хочет ли  участник команды получить новости.  это быстрее, намного быстрее. web socket делает тоже самое, только в пределах сети, web socket  позволяет в реальном времени отправлять сообщения при этом не ожидая что клиент прямо сейчас отправит ответ


# URL 
web socket как и http использует порт 80 или 443,  в зависимости от того, является ли соединение безопасным. также если web socket использует tls то его название будет wss

### примеры
-  заходим на сайт по протоколу ws с портом 80: ws://www.yandex.ru:80
- заходим на сайт по протоколу ws с портом 443: ws://www.google.com:443
- заходим на сайт по  wss с tls соединением: wss://www.microsoft.com


# установка соединения

### установка соединения - запрос клиента

установка соединения в ws  очень  похожа на http, но есть пара важных нюансов которые следует показать:

```http
GET /chat/tg HTTP/1.1
Host: www.telegram.org.
Upgrade: websocket //Upgrade нужен для того, чтобы указать серверу что устанавливается web socket соединение
Connection: Upgrade // указывает какое соединение мы хотим совершить
Sec-WevSocket: dGhI7JVXvZSDb25LlIjZQ== // это 16 байт сгенерированых случайным образов в base64, он нужен для того чтобы определить, поддерживает ли сервер веб сокет
Origin: http://telegram.org.
Sec-WebSocket-Version: 13 // указывает текущую версию вебсокетов
```

### установка соеддинения - ответ сервера

после отправки запроса, клиент получит что то такое если сервер  поддерживает веб сокеты:

```http
HTTP/1.1 101 Swiching Protocols // указывает что протокол http   заменен на веб сокет
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Accept: s3pLY+x))))Zbk+xOo= // нужен чтобы сказать клиенту то, что сервер поддерживает веб сокеты

```



# передача данных

- данные передаются в виде фреймов(кадров) через tcp сокет
- заголовки фреймов имеют бинарный вид,  так как это улучшает производительность и уменьшает накладные расходы
-  также в результате передачи данных может быть **Фрагментация**. это - большой фрейм делится на несколько маленьких и все эти маленькие фреймы передаются 
- также у веб сокетов есть несколько типов кадров:
1. текстовые данные в UTF-8(не рекомендуется)
2. бинарные данные
3. управляющие фреймы(нужны для передачи какого то сигнала, тоесть для поддержания или закрытия соединения)

#### управляющие фреймы
- ping-pong : это когда сервер через определенные промежутки времени посылает ping, ожидая что клиент выдаст pong, если по истечению тайм аута pong не придет,  соединение закрывается
- close - используется для разрыва соединения. соединение может закрыть любая сторона когда она захочет.




# заголовок

![[frame_header_struct.png]]

- opcode - говорит о том какой тип у фрейма(close,ping,бинарные данные, текстовые данные и тд)
- данные фрейма(playload) - содержит данные которые нужно передать
- длина тела(playload len) - содержит длину playload если фрейм в пределах 7 бит,  если данные больше то могут использоваться дополнительные биты. может использоваться расширенная длина байт.  
- FIN - используется для указания фрагментации, если флаг =1 то значит что кадр последний. при этом если данные очень маленькие и поместились в 1  фрейм, то FIN =1. также во всех кадрах кроме последнего FIN = 0(если фрейм был фрагментирован)
- RSV1-3 - зарезирвированы и нужны для расширения протокола. клиент и сервер догавариваются как использовать эти флаги
- MASK - указывает является ли фрейм масикрованным, 1 если маскирован, иначе 0
- MASKING KEY - есть только если MASK = 1. генерируется отправителем и для каждого фрейма он разный
