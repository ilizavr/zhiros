# работа ARP 
- когда хост должен получить MAC адрес по IP, он делает ARP запрос примерно так:
    ARP-запрос (Request)

        Хост, которому нужно узнать MAC по IP (например, для 192.168.1.2), отправляет broadcast-запрос (на MAC FF:FF:FF:FF:FF:FF) с вопросом:
        "У кого IP 192.168.1.2? Сообщите ваш MAC!"

        В запросе указывается:

            Sender MAC: свой MAC (например, A1:FF:C4:FF:FA:DC).

            Sender IP: свой IP (например, 192.168.1.1).

            Target IP: 192.168.1.2.

            Target MAC: 00:00:00:00:00:00 (так как он неизвестен).

    ARP-ответ (Reply)

        Хост с IP 192.168.1.2 отвечает не broadcast-ом, а напрямую (unicast) на MAC A1:FF:C4:FF:FA:DC, сообщая свой MAC.

        В ответе указывается:

            Sender MAC: свой MAC (например, B2:EE:D3:AA:BB:CC).

            Sender IP: 192.168.1.2.

            Target MAC: A1:FF:C4:FF:FA:DC (из запроса).

            Target IP: 192.168.1.1 (из запроса).

# ARP таблица(кэш)

ARP-кэш — это временная таблица, хранящаяся в памяти устройства (компьютера, роутера и т. д.), которая содержит соответствия **IP-адресов → MAC-адресов** в локальной сети.

#### **Как работает ARP-кэш?**

1. Когда устройству нужно узнать MAC по IP, оно сначала проверяет свою ARP-таблицу.
    
2. Если записи нет, отправляется **ARP-запрос**, и после ответа новая пара **IP → MAC** сохраняется в кэше.
    
3. Записи имеют **время жизни (TTL)**, обычно **15–30 минут**, после чего удаляются (если не обновляются).
    
4. Если устройство не отвечает на ARP-запросы, его запись может быть удалена раньше.

#### пример ARP кэша
| Интернет-адрес  | Физический адрес  | Тип          |
| --------------- | ----------------- | ------------ |
| 192.168.1.1     | 00-11-22-33-44-55 | динамический |
| 192.168.1.100   | aa-bb-cc-dd-ee-ff | статический  |
| 224.0.0.22      | 01-00-5e-00-00-16 | статический  |
| 255.255.255.255 | ff-ff-ff-ff-ff-ff | статический  |


Записи ARP-таблицы бывают двух вид видов: статические и динамические. Статические добавляются самим пользователем, динамические же – создаются и удаляются автоматически. При этом в ARP-таблице всегда хранится широковещательный физический адрес FF:FF:FF:FF:FF:FF (в Linux и Windows).




# прокси ARP(proxy ARP)

**Proxy ARP (ARP-прокси)** — это механизм, при котором маршрутизатор (или иное устройство) **отвечает на ARP-запросы за другие хосты**, даже если они находятся в другой IP-подсети. Это нарушает классическое поведение ARP, где ответ должен приходить только от владельца запрашиваемого IP.

**Исторический контекст**:

- Появился в ранних IPv-сетях для совместимости с хостами, **не знающими о маршрутизации** (например, без настроенного шлюза по умолчанию).
    
- Описан в RFC 1027 (1987) как метод "транзитного ARP".


#### **1. Определение и контекст**

**Proxy ARP (ARP-прокси)** — это механизм, при котором маршрутизатор (или иное устройство) **отвечает на ARP-запросы за другие хосты**, даже если они находятся в другой IP-подсети. Это нарушает классическое поведение ARP, где ответ должен приходить только от владельца запрашиваемого IP.

**Исторический контекст**:

- Появился в ранних IPv-сетях для совместимости с хостами, **не знающими о маршрутизации** (например, без настроенного шлюза по умолчанию).
    
- Описан в RFC 1027 (1987) как метод "транзитного ARP".
    

---

#### **2. Принцип работы (пошагово)**

Рассмотрим сеть с тремя устройствами:

- **Хост A**: `192.168.1.10/24`, шлюз не указан.
    
- **Роутер R**: интерфейсы `192.168.1.1/24` и `192.168.2.1/24`.
    
- **Хост B**: `192.168.2.20/24`.
    

**Сценарий**:  
Хост A пытается отправить пакет хосту B, **ошибочно полагая**, что тот находится в той же подсети (из-за неверной маски или настройки).

##### **Шаг 1. Проверка подсети на хосте A**

Хост A вычисляет, принадлежит ли `192.168.2.20` его подсети (`192.168.1.0/24`):

- Применяет маску:
    
    
    

- 192.168.1.10 AND 255.255.255.0 = 192.168.1.0  
    192.168.2.20 AND 255.255.255.0 = 192.168.2.0  
    
- Подсети не совпадают → **обычный хост должен отправить пакет шлюзу**.
    
- **Но!** Если шлюз не настроен (или маска `/16`), хост A решает, что B в его сети.
    

##### **Шаг 2. ARP-запрос от A**

Хост A отправляет **broadcast ARP-запрос**:

text

Sender MAC: AA:AA:AA:AA:AA:AA  
Sender IP: 192.168.1.10  
Target MAC: 00:00:00:00:00:00  
Target IP: 192.168.2.20  

##### **Шаг 3. Ответ роутера R (Proxy ARP)**

Роутер R, видя, что:

- `192.168.2.20` доступен через другой его интерфейс,
    
- **отвечает своим MAC-адресом**, как если бы он был владельцем `192.168.2.20`:
    

text

Sender MAC: RR:RR:RR:RR:RR:RR (MAC интерфейса 192.168.1.1)  
Sender IP: 192.168.2.20  # Подмена!  
Target MAC: AA:AA:AA:AA:AA:AA  
Target IP: 192.168.1.10  

##### **Шаг 4. Передача пакета**

- Хост A отправляет пакет на MAC `RR:RR:RR:RR:RR:RR` (думая, что это B).
    
- Роутер R перенаправляет пакет в сеть `192.168.2.0/24` через обычную маршрутизацию.
    

---

#### **3. Технические детали**

##### **Условия срабатывания Proxy ARP**

1. Роутер знает маршрут к целевой подсети (есть в таблице маршрутизации).
    
2. **Отключена проверка исходной подсети** (обычно роутер игнорирует ARP для "чужих" подсетей).
    

##### **Формат ARP-пакета при Proxy ARP**

|Поле|ARP-запрос от A|ARP-ответ от R (Proxy)|
|---|---|---|
|Sender MAC|AA:AA:AA:AA:AA:AA|RR:RR:RR:RR:RR:RR|
|Sender IP|192.168.1.10|192.168.2.20 (подмена!)|
|Target MAC|00:00:00:00:00:00|AA:AA:AA:AA:AA:AA|
|Target IP|192.168.2.20|192.168.1.10|

##### **Поведение по стандартам**

- RFC 1027: Proxy ARP — это **"услуга" (service)**, а не обязательная функция.
    
- RFC 2002: В мобильном IP Proxy ARP используется для перенаправления трафика к мобильным хостам.




# добровольный ARP(gratuitous ARP,GARP)

#### базовое определение
добровольный ARP запрос(GARP REQUEST),  это запрос при котором хост запрашивает свой собственный IP-адрес  с целью  обновить соответствие адресов в ARP кэше. если же приходит добровольный ARP ответ(GARP REPLY) то это значит что текущий IP уже занят. 

#### как работает

#### ожидаемый кейс: полноценный GARP

Когда устройство получает новый IP (например, при подключении к сети или смене адреса), оно должно отправить GARP-запрос, чтобы уведомить все хосты в сети о новом соответствии IP-MAC. Это гарантирует, что:

- Все соседи обновят свои ARP-таблицы
    
- все обновят Старые записи (если IP раньше принадлежал другому устройству) будут перезаписаны
    

#### **Мой кейс: ленивый GARP**

Я уехал на 10 дней, мой старый IP (`192.168.0.10`) освободился, но умный чайник от Яндекс с поддержкой Алисы, которое получило этот IP, **не отправило GARP**. В результате:

- В ARP-кэшах других устройств остался **мой старый MAC-адрес**
    
- Пакеты на `192.168.0.10` какое-то время уходили "в никуда"
    
- Только после таймаута ARP-кэша (от нескольких минут до часов) сеть начала работать корректно
    

**Вывод:** GARP критически важен для мгновенного обновления ARP-записей. Если устройство его не отправляет — возможны временные проблемы с сетью! и еще, если чайник так делает, выкиньте его и купите hub, он хотя бы фреймы раздавать умеет!

RARP  - это протокол который делает тоже самое что и обычный ARP, но он находит IP по MAC, а не как обычный ARP который находит MAC по IP. в настоящее время выкинут на помойку потому что RARP это как тупой помощник -  вроде он дал то что надо, но выйти в сеть ты все равно нихера не можешь.
